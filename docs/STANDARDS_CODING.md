# コーディング標準（TDD・スケルトン駆動）

**参照元**: CLAUDE.md → `@docs/STANDARDS_CODING.md`
**詳細アーカイブ**: [CONSTITUTION_DETAIL.md](CONSTITUTION_DETAIL.md)

---

## 1. 実装フロー（厳守）

```
[1] スケルトン作成
     └─ クラス/関数/分岐/変数 + WHYコメント
        ↓
[2] 単体テスト作成
     └─ 正常系 / 異常系 / 境界値
        ↓
[3] 実装（テストが通るまで）
        ↓
[4] リファクタ + カバレッジ確認（80%以上）
        ↓
[5] ドキュメント/関連ファイル更新
```

## 2. コメント原則

| 書く | 書かない |
| --- | --- |
| WHY：なぜその分岐か | WHAT：コードが説明できること |
| 入出力の制約・前提条件 | 関数・変数の一覧 |
| 非自明なアルゴリズム | 「used by X」「added for Y」 |
| 境界値の扱い | 過去のタスク参照 |

## 3. テンプレート

### 関数

```text
// [機能名]
// # 入力: param1（型、制約）, param2（型、制約）
// # 出力: 戻り値（型、null/error の扱い）
// # 前提: [前提条件があれば]
```

### 分岐

```text
// [分岐の意図]: なぜこの条件が必要か
if condition {
    // [この分岐で何をするか、なぜか]
} else {
    // [else 分岐の意図]
}
```

## 4. カバレッジ要件

- **目標**: 行カバレッジ 80% 以上
- **優先度**: 制御フロー（分岐）カバレッジ > 行カバレッジ
- **ビジネスロジック境界値**: 100% を目指す

## 5. 成果物完全性チェック

コード修正時は以下**すべて**更新する：

- [ ] 実装そのもの（型安全性・パフォーマンス）
- [ ] 既存テストの有効性確認 + 新規テスト追加
- [ ] API仕様・使用例ドキュメント
- [ ] スケルトンコメントとの一貫性
- [ ] 参照元・依存ファイルの整合性

## 6. 禁止事項

- コードだけ修正してテスト未更新
- 仕様変更してドキュメント放置
- スケルトンコメントと実装の乖離
- 既存テストの削除/スキップ化（[[STOP条件]]）
