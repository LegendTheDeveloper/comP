# トークンを節約するVSCode拡張機能「comP」を作った話

## きっかけ：愛用ツールの有料化

AI コーディングを日常的に使っている人なら、コンテキストウィンドウとトークン消費量がどれだけ痛いか分かると思う。そこで重宝していた、コードをうまく圧縮してトークンを削ってくれる拡張機能があったんだけど、ある日突然「無料プランは月 ×× 回まで」という表示が出た。

「……まあ、そういうもんだよな」と思いつつも、釈然としない。

そこで気になったのが「あれって、どういう仕組みでやってるんだろう？」ということ。

---

## どうやってトークンを圧縮しているのか調べてみた

調べてみると、主なアプローチはいくつかある。

### 手法①：AST ベース圧縮

tree-sitter などの構文解析器でコードをパースし、コメントや空行を除去したり、関数ボディを `{ ... }` に置き換えて「シグネチャだけ」を残す手法。

```rust
// Level 1: compact（コメント・空行除去）
fn authenticate(user: &User) -> Result<Token> {
    // ← コメントはここで消える
    validate(user)?;
    generate_token(user)
}

// Level 2: skeleton（ボディ全部消す）
fn authenticate(user: &User) -> Result<Token> { ... }
```

実装すれば 50〜80% のトークン削減になる。

### 手法②：シンボルグラフ + BM25 検索

コードベース全体を SQLite グラフ DB にインデックスして（ノード＝シンボル、エッジ＝依存関係）、AI が「今必要なファイルだけ」を取得できるようにする。無関係なファイルを読ませない、というアプローチ。

```
タスク「認証バグを修正して」
     ↓ キーワード分解
  ["認証", "auth", "token"]
     ↓ グラフ検索（TF-IDF + LIKE）
  src/auth.rs, src/jwt.rs, src/middleware.rs  ← これだけ返す
     ↓
  全 200 ファイルを読ませる必要なし
```

「これなら自分でも作れるんじゃないか？」

## 作ってみることにした

### 構成

調べた結果、こういう構成にした。

```
comP/
├── src/              ← VSCode 拡張（TypeScript）
│   ├── daemon/       ← Rust デーモンとの通信層
│   └── ui/           ← CodeLens、ステータスバー
├── daemon/           ← 検索エンジン本体（Rust）
│   └── src/
│       ├── indexer/  ← tree-sitter でシンボル抽出
│       ├── graph/    ← SQLite グラフ DB
│       ├── search/   ← TF-IDF + BM25
│       └── mcp/      ← JSON-RPC 2.0 サーバー
└── resources/
```

VSCode 拡張（TypeScript）がフロントエンドで、実際の重い処理は Rust デーモンが担当。両者は `stdin/stdout` で JSON-RPC 2.0 を使って通信する。いわゆる MCP（Model Context Protocol）という仕組みで、Claude Code や Cursor などのエージェントとも直接連携できる。

### 言語は Rust を選んだ（生成AIに書いてもらった）

正直に言う。Rust はほとんど書いたことがない。

でも「コード解析ツールなんだから、速くてメモリ安全なほうがいいよな」という理由で Rust にした。実際に自分が書いたのは仕様とアーキテクチャの設計で、実装の大部分は生成 AI に任せた。

ただ、「生成 AI が書いたから」と全部そのまま使う気にはなれなかった。一通りレビューして、変だと思ったところは直した。後述するが、ソースを人様に見せるとなると、それなりにちゃんとしたものにしたかったのだ。

## VSCode 拡張機能の作り方、まず何も分からなかった

「`yo code` でスキャフォールドして、`vsce package` で VSIX にして、ローカルは `code --install-extension` でインストールできる」

……そこまでは分かった。

問題は **マーケットプレイスへの公開**。

### Azure DevOps まわりで迷子になった

VSCode マーケットプレイスに拡張機能を出すには、まず **Azure DevOps** で Publisher を作って、**Personal Access Token（PAT）** を発行して、それを `vsce publish` に渡す。

「Azure？なんで Marketplace に登録するのに Azure が出てくるの？」

画面遷移が独特すぎて最初は完全に迷子。「Organization」を作れと言われてもピンとこない。PAT を発行する画面がどこにあるか探すのに 30 分かかった。

さらに GitHub Actions で自動リリースを組もうとしたら、今度は GitHub の Secrets と Azure PAT の連携でハマる。

```yaml
# .github/workflows/release.yml
- name: Publish to VSCode Marketplace
  run: vsce publish -i comp-vscode-${{ steps.version.outputs.tag }}.vsix
  env:
    VSCE_PAT: ${{ secrets.VSCE_PAT }}
```

こういうの、慣れてる人には当たり前なんだろうけど、最初は本当にわからない。Azure の画面がモッサリしてるのも相変わらずで（昔から変わってないな、あれ……）、結局試行錯誤で 2〜3 時間溶けた。

## ソースを晒すのが恥ずかしい問題

若い頃、フリーソフトを作って公開したことはある。でも「バイナリだけ」配布で、ソースを見せたことはなかった。会社でもいろいろツールを作ってきたが、社内に閉じていた。

今回、OSS として GitHub で公開するということは、全コードが世界中に見える。

「生成 AI が書いたんだけど……」と言っても、リポジトリのオーナーは自分だ。恥ずかしいものは恥ずかしい。

なので多少時間をかけて：

- 命名がヘンなところを直す
- 不要なコメントを整理する
- テストが弱いところを補強する
- `cargo clippy` の警告をゼロにする

ひと通りやった。いや、完璧ではないんだけど、「まあこれくらいなら人に見せられるか」という水準には持っていった気がする（たぶん）。

## コーヒー一杯分の寄付ボタンを設置しようとしてハマった

「せっかく公開するなら、Ko-fi か Buy Me a Coffee でコーヒー一杯分の寄付ボタンでも置いてみよう」と思った。軽い気持ちで。

…甘かった。

登録を進めると「あなたは日本在住ですか？ならば米国税法上の源泉徴収のため **W-8BEN フォーム**への記入が必要です」と出てくる。

W-8BEN とは何か。外国人が米国源泉所得を受け取る際、租税条約に基づいて源泉徴収税率の軽減を申請するための書類だ。個人情報（住所・TIN: 納税者番号）を英語で記入して提出する必要がある。

「コーヒー代 ¥500 のために何をさせるんだ……」

結局手続きは終わらせたが、コーヒー代より手続きの手間の方がだいぶ重かった。寄付プラットフォームを使おうとしている日本在住の方はご注意を。

## 完成して使ってみた感想

`run_pipeline` ツールを Claude Code や Cursor に登録すると、こんな感じで動く。

```
▶ run_pipeline({ "task": "JWT の有効期限バグを直して" })

→ pivot_files:
  - src/auth/jwt.rs       (12 symbols, 600 tokens)
  - src/middleware/auth.rs (8 symbols, 400 tokens)
→ savings: 68%
→ estimated_cost: $0.03
```

全ファイルをエージェントに読ませると 74,000 トークン超えるところが、関連ファイルだけ絞ると 1,000 トークン程度で済む。使い込んでいるうちに「あ、これ地味に効いてるな」と感じるようになった。

あと、PDF や Word ファイルまでインデックスされて検索対象になるのが意外と便利だった。仕様書や設計書もコードと一緒に検索できる。
