# TDD-Flow スキル

## 起動条件

- `skeleton-first` 完了後の自動継続
- 既存コードのテスト追加依頼
- 「TDDで実装」の明示指示

## 実行手順

### 1. テストケース洗い出し

スケルトンの分岐・境界値から、以下3カテゴリでケース作成：

| カテゴリ | 例 |
| --- | --- |
| 正常系 | 期待通りの入力 → 期待通りの出力 |
| 異常系 | null/undefined、不正な型、範囲外 |
| 境界値 | 0, 1, 最大値, 最大値-1, 最大値+1 |

### 2. テスト先行作成

`src/<module>/tests/<feature>_test.<ext>` に作成。

**テスト命名規則**：
```
describe('FeatureName', () => {
  describe('methodName', () => {
    test('正常系: [入力] → [期待値]', () => {});
    test('異常系: [異常入力] → [期待例外]', () => {});
    test('境界値: [境界条件] → [期待挙動]', () => {});
  });
});
```

### 3. テスト実行（全FAIL確認）

```bash
# まず全てfailすることを確認（テストが正しく機能している証拠）
npm test -- <feature>_test
# または
pytest src/<module>/tests/<feature>_test.py -v
```

### 4. 実装

スケルトンのTODOを埋めて、**1テストずつPASSさせる**。
全テストGREENまで反復。

### 5. カバレッジ確認

```bash
npm test -- --coverage
# または
pytest --cov=src --cov-report=term-missing
```

**基準**:
- 行カバレッジ 80% 以上
- 分岐カバレッジを優先確認
- ビジネスロジック境界値は100%目標

### 6. リファクタ

テストGREENを維持しながら、コード品質改善：
- 重複削除
- 命名改善
- スケルトンコメントと実装の一貫性確認

### 7. 完了報告

ユーザーに以下を報告：
- 追加テスト数（正常/異常/境界値の内訳）
- カバレッジ数値
- 既存テストへの影響有無

## 禁止事項

- テストなしでの実装
- テストを「失敗するから」削除・スキップ化（STOP条件#3）
- カバレッジ未確認での完了宣言
- モックを多用した「中身を見ないテスト」
