# Skeleton-First スキル

## 起動条件

- 新規機能・新規モジュールの実装依頼
- 「〇〇を実装して」「〇〇クラスを追加」等の指示
- リファクタで構造から作り直す場合

## 実行手順

### 1. 要件整理

ユーザーに以下を確認（不明点があれば必ず聞く）：

- 機能の目的・責務
- 入出力の仕様
- 制約条件・前提条件
- 既存コードとの関係（vexp `run_pipeline` で関連箇所を取得）

### 2. スケルトン作成

`temp/skeleton/<機能名>_skeleton.<ext>` に以下を生成：

```text
// [機能名]
//
// # 責務
// - [責務1]
// - [責務2]
//
// # 公開インターフェース
// - [メソッド名]: [目的]

class FeatureName {
    // [プロパティの用途と制約]
    private field: Type;

    // [メソッド名]
    // # 入力: [param説明]
    // # 出力: [戻り値説明]
    // # 前提: [前提条件]
    // # 例外: [発生しうる例外]
    methodName(param: Type): ReturnType {
        // [分岐の意図]: なぜこの条件判定か
        if (condition) {
            // TODO: [この分岐で何をするか]
        } else {
            // TODO: [else 分岐の意図]
        }
    }
}
```

### 3. ユーザー確認

スケルトンを提示し、**実装前に承認を得る**：

「スケルトンを `temp/skeleton/xxx_skeleton.ts` に作成しました。設計意図を確認してから、テスト作成に進みます。問題ありませんか？」

### 4. 完了後

承認後、`tdd-flow` スキルへ引き継ぐ。

## 禁止事項

- スケルトン作成をスキップして実装に進む
- WHYコメントなしの空骨格を「スケルトン」と称する
- ユーザー承認なしにテスト・実装に進む
